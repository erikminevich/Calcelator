# spec.md — Спецификация требований к приложению-калькулятору

## 1. Назначение и контекст
Цель проекта: разработать консольное приложение-калькулятор с поэтапным расширением функциональности (Stage 1-3) и обязательным автоматизированным тестированием.

Источник требований:
- task.md (формулировка лабораторной);
- Требования Чек-лист.pdf.

## 2. Границы системы
В систему входит:
- парсер арифметических выражений;
- вычислитель выражений;
- CLI-интерфейс приложения;
- автоматические тесты и отчеты по тестам;
- нагрузочные измерения и benchmark-отчет.

В систему не входит:
- web-интерфейс;
- распределенные вычисления;
- выполнение пользовательского кода из выражений.

## 3. Поэтапная поставка и управление Git
Обязательные этапы разработки:
1. stage1: базовые операции + - * /, числа с плавающей точкой, unit-тесты парсера и вычислителя.
2. stage2: научная нотация, ^, скобки, интеграционные тесты.
3. stage3: функции sqrt/sin/cos/tg/ctg/ln/exp, константы pi/e, режим градусов/радиан, функциональные и нагрузочные тесты.

Правила контроля версий:
- каждому этапу соответствует минимум один отдельный коммит;
- каждому этапу соответствует git-тег (stage1, stage2, stage3);
- итоговая история должна позволять проверить любой этап переключением по тегу.

## 4. Функциональные требования

### FR-1. Ввод данных
- Способ ввода: CLI-аргументы.
- Источник: пользователь командной строки/скрипт CI.
- Точность: float64 (как минимум).
- Диапазон: корректные арифметические выражения в текстовом виде.
- Частота ввода: не ограничена, запуск приложения многократный.

### FR-2. Вывод данных
- Способ вывода: stdout/stderr CLI.
- Назначение: получение результата вычисления или диагностической ошибки.
- Формат:
  - успех: десятичный или экспоненциальный формат числа;
  - ошибка: текст ошибки с указанием типа (парсинг/вычисление).
- Частота вывода: один результат или одна ошибка на один запуск.

### FR-3. Форматы отчетов
- Форматы для отчетов по тестам: консольная таблица в стиле Pretty Table + текстовый файл отчета в репозитории.
- Форматы Web-страниц: не применимо (web-интерфейс отсутствует).

### FR-4. Внешние интерфейсы
- Программные интерфейсы:
  - модуль парсинга (parse);
  - модуль вычисления (evaluate);
  - общий API (calculate);
  - CLI (calc).
- Аппаратные интерфейсы: не требуются.

### FR-5. Внешние коммуникационные интерфейсы
- Сетевые интерфейсы и протоколы отсутствуют.
- Соединения с удаленными сервисами не используются.

### FR-6. Задачи пользователя
Пользователь должен иметь возможность:
1. Ввести арифметическое выражение и получить результат.
2. Получить понятное сообщение об ошибке, если выражение некорректно или невычислимо.
3. Запустить полный набор автотестов одной командой.
4. Получить табличный лог тест-кейсов для текущего этапа.

### FR-7. Данные по задачам пользователя
- Входные данные: строка выражения и (на Stage 3) CLI-флаг единицы угла.
- Выходные данные:
  - числовой результат;
  - код возврата (0 при успехе, не 0 при ошибке);
  - тестовый отчет с колонками Input, Expected, Actual, Message.

## 5. Нефункциональные требования

### NFR-1. Временные параметры
- Stage 3: обработка выражений длиной до 1000 символов не более 200 мс (ориентир из задания).
- Для более длинных выражений выполнение должно оставаться устойчивым (без аварийного завершения).

### NFR-2. Надежность
- Ошибки парсинга и вычисления различаются и возвращаются корректно.
- Невозможные операции (например, деление на ноль, численное переполнение) возвращают контролируемую ошибку.
- Неподдерживаемые выражения не должны попадать в вычислитель.

### NFR-3. Защищенность
- Запрещено вычисление через eval/exec и аналогичные механизмы выполнения произвольного кода.
- Система должна обрабатывать пользовательский ввод только как данные.

### NFR-4. Память и диск
- Приложение должно работать в пределах типичных ресурсов учебной машины.
- Дополнительные артефакты тестов и отчетов должны иметь разумный размер и храниться в проекте структурированно.

### NFR-5. Сопровождаемость и адаптация
- Код разбит на независимые модули: парсер, AST, вычислитель, CLI, тесты.
- Структура проекта должна поддерживать поэтапное развитие без ломки прошлых этапов.

### NFR-6. Критерии успеха/неуспеха
Успех:
- выполнены требования всех стадий;
- все тесты проходят одной командой;
- история git содержит раздельные этапы и теги.

Неуспех:
- отсутствует поэтапность коммитов;
- тесты не автоматизированы или нестабильны;
- требования stage не покрыты реализацией/тестами.

## 6. Критерии приемки и тестируемость
Система считается принятой, если:
1. Для каждого stage есть отдельный коммит и тег.
2. Все обязательные тесты задания автоматизированы.
3. Добавлены дополнительные тесты: суммарно не менее 15 (распределение по stage допускается неравномерное).
4. Тесты и отчеты запускаются одной командой.
5. Формируется табличный лог кейсов с ожидаемым/фактическим результатом.

## 7. Качество требований
- Требования сформулированы понятным языком для пользователя и разработчика.
- Внутренние противоречия не допускаются.
- Баланс параметров-антагонистов:
  - приоритет корректности и надежности выше микрооптимизаций;
  - приоритет тестируемости выше сокращения объема кода.
- Каждое требование должно быть трассируемо к исходному заданию или чек-листу.
- Каждое требование должно быть проверяемо независимым тестом.

## 8. Полнота требований
- До начала разработки зафиксированы минимально необходимые ограничения stage 1-3.
- Неопределенности (точные стенды benchmark и абсолютные значения времени) фиксируются в отчетах измерений.
- Требования считаются полными, если после их выполнения приложение пригодно для сдачи лабораторной и проверки по тегам.

## 9. Вероятные изменения требований
- Добавление новых математических функций: средняя вероятность.
- Ужесточение порогов по производительности: средняя вероятность.
- Расширение CLI-интерфейса (новые флаги): средняя вероятность.
- Изменение формата отчетов: высокая вероятность.
